== Назначение документа

Пошаговая инструкция по развёртыванию link:https://github.com/mbolli/nfsen-ng[nfsen-ng] на чистой *Ubuntu 24.04 LTS* (Server или Desktop) через *Docker*.

Документ рассчитан на сценарий, когда NetFlow/IPFIX поступает *с конечных хостов* (Windows/Ubuntu-агенты), а не с сетевого оборудования. Коллектор и веб-интерфейс — на одном Linux-сервере.

== Архитектура

[source,text]
----
  win-pc01 (netflowAgent)  ──NetFlow v9 UDP :9995──►  nfcapd → live/win-pc01/
  ubuntu-pc01 (агент)      ──NetFlow v9 UDP :9996──►  nfcapd → live/ubuntu-pc01/
                              │
                              ▼
                    nfsen-ng (Docker) → браузер :9000
----

[IMPORTANT]
====
* *nfsen-ng не принимает NetFlow/IPFIX по сети* — это только визуализатор поверх nfdump.
* *nfcapd обязателен на коллекторе* — он принимает UDP-поток и пишет файлы `nfcapd.*`.
* На endpoint-хостах nfcapd *не нужен* — только агент, отправляющий flow на коллектор.
====

=== Компоненты

[cols="1,2,2", options="header"]
|===
| Компонент | Где работает | Роль

| netflowAgent (Rust)
| Windows / Ubuntu endpoint
| Сбор трафика, экспорт NetFlow v9 по UDP

| `nfcapd`
| Ubuntu-коллектор (хост, не Docker)
| Приём flow, запись в `/var/nfdump/profiles-data/live/<source>/`

| `nfsen-ng`
| Ubuntu-коллектор (Docker)
| Графики (RRD), Flows/Statistics через nfdump, веб-UI

|===

== Требования

=== Сервер (коллектор)

* Ubuntu 24.04 LTS (чистая установка)
* 2 vCPU, 4 GB RAM (минимум для лаборатории)
* 40+ GB диск (растёт с объёмом flow-данных)
* Сеть: статический или DHCP IP, доступный агентам по UDP

=== Порты

[cols="1,1,2", options="header"]
|===
| Порт | Протокол | Назначение

| 9995 (и др.)
| UDP
| Приём NetFlow/IPFIX (`nfcapd`)

| 9000
| TCP
| Веб-интерфейс nfsen-ng (HTTP)

|===

== Шаг 1. Подготовка системы

Откройте терминал и выполните:

[source,bash]
----
sudo apt update && sudo apt upgrade -y
sudo apt install -y curl git ca-certificates nano
----

Узнайте IP сервера (понадобится для настройки агентов):

[source,bash]
----
ip -4 addr show
----

== Шаг 2. Установка Docker

[source,bash]
----
curl -fsSL https://get.docker.com | sudo sh
sudo usermod -aG docker $USER
----

[IMPORTANT]
====
После `usermod` выполните *полную перезагрузку* системы (`sudo reboot`). Закрытия терминала или блокировки экрана недостаточно — группа `docker` подхватывается только в новой сессии входа.

После reboot проверьте:

[source,bash]
----
groups
# в списке должна быть docker
----
====

Проверка:

[source,bash]
----
docker --version
docker compose version
docker run --rm hello-world
----

== Шаг 3. Установка nfcapd (коллектор flow)

[source,bash]
----
sudo apt install -y nfdump
which nfcapd
nfcapd -V
----

image::images/nfcapd-version.png[Nfcapd version,align="center"]

=== Каталоги и источники (win-pc01 + ubuntu-pc01)

[IMPORTANT]
====
* *Каталог сам по себе ничего не собирает* — `mkdir` только создаёт место на диске.
* *nfcapd* принимает UDP и *пишет* файлы в каталог из `-w`.
* *nfsen-ng* только *читает* каталоги, перечисленные в `NFSEN_SOURCES`; flow туда не перенаправляет.

Один UDP-порт = один процесс `nfcapd`. Чтобы разделить два хоста в UI, каждому — свой порт и своя папка под `live/`.
====

В проекте netflowAgent — *два конечных устройства*:

[cols="1,1,1,2", options="header"]
|===
| Хост | UDP-порт на коллекторе | Каталог на диске | Имя в nfsen-ng

| win-pc01
| 9995
| `/var/nfdump/profiles-data/live/win-pc01/`
| `win-pc01`

| ubuntu-pc01
| 9996
| `/var/nfdump/profiles-data/live/ubuntu-pc01/`
| `ubuntu-pc01`

|===

Создайте каталоги:

[source,bash]
----
sudo mkdir -p /var/nfdump/profiles-data/live/{win-pc01,ubuntu-pc01}
----

=== Запуск nfcapd (два источника)

[source,bash]
----
sudo nfcapd -w /var/nfdump/profiles-data/live/win-pc01    -z=lz4 -S 1 -p 9995 -D
sudo nfcapd -w /var/nfdump/profiles-data/live/ubuntu-pc01 -z=lz4 -S 1 -p 9996 -D
----

[cols="1,3", options="header"]
|===
| Флаг | Значение

| `-w`
| Каталог записи; путь `.../live/<source>` задаёт имя источника

| `-S 1`
| Структура `YYYY/MM/DD/` — *обязательно* для nfsen-ng

| `-z=lz4`
| Сжатие capture-файлов

| `-p 9995`
| UDP-порт приёма NetFlow/IPFIX

| `-D`
| Запуск в фоне (daemon)

|===

[NOTE]
====
В nfcapd 1.7.x флаг `-T all` *устарел* (`Option -T no longer supported and ignored`). В командах выше его нет — расширения flow захватываются по умолчанию.
====

[TIP]
====
Папка `live/all` в этой схеме *не используется*. Если все агенты шлют на один порт, flow смешивается в одном каталоге — для двух хостов так делать не нужно.
====

=== Firewall (ufw)

На *Ubuntu Desktop* `ufw` часто *выключен* по умолчанию — тогда порты на VM уже доступны из LAN (если их не блокирует корпоративный firewall).

Сначала проверьте статус:

[source,bash]
----
sudo ufw status verbose
# или кратко:
sudo ufw status
----

[cols="1,2", options="header"]
|===
| Вывод | Что делать

| `Status: inactive`
| *Ничего не обязательно.* UDP 9995/9996 и TCP 9000 не фильтруются ufw. Переходите к docker-compose. Правила ниже — только если решите включить ufw позже.

| `Status: active`
| Добавьте правила *до* или *после* enable — см. ниже.

|===

Если ufw *активен* или вы хотите его включить:

[source,bash]
----
sudo ufw allow 9995/udp   # win-pc01
sudo ufw allow 9996/udp   # ubuntu-pc01
sudo ufw allow 9000/tcp   # nfsen-ng UI
sudo ufw enable           # только если ufw был inactive и вы сознательно включаете его
sudo ufw status numbered
----

[WARNING]
====
`sudo ufw enable` на машине с `inactive` *включает* firewall и начинает блокировать всё, что не разрешено явно (по умолчанию deny incoming). Не запускайте `enable`, если ufw выключен и вас устраивает текущее поведение.
====

[TIP]
====
Корпоративный межсетевой экран между endpoint и VM ufw не заменяет — даже при `inactive` на Ubuntu трафик может блокироваться *вне* вашей VM.
====

=== Проверка приёма данных (необязательно на этом этапе)

[NOTE]
====
*Perl-агенты netflowAgent на этом шаге **не нужны**.* Достаточно того, что `nfcapd` слушает порты 9995 и 9996 (проверка через `ss -ulnp`).

Пустые каталоги `live/win-pc01/` и `live/ubuntu-pc01/` — *нормально*. Файлы `nfcapd.*` появятся только когда:

* заработают ваши агенты (win-pc01 → :9995, ubuntu-pc01 → :9996), или
* вы временно проверите приём через <<тест-softflowd,softflowd>> на самой VM.

*Сейчас можно переходить к шагу 4* (docker-compose / nfsen-ng). Initial Import в UI имеет смысл после появления хотя бы одного flow-файла.
====

Когда агенты (или softflowd) уже шлют данные — подождите 5–10 минут (ротация файлов) и выполните:

[source,bash]
----
find /var/nfdump/profiles-data/live/win-pc01 -name "nfcapd.*" -ls
find /var/nfdump/profiles-data/live/ubuntu-pc01 -name "nfcapd.*" -ls
----

== Шаг 4. Загрузка и настройка docker-compose

[source,bash]
----
mkdir -p ~/nfsen-ng && cd ~/nfsen-ng
curl -O https://raw.githubusercontent.com/mbolli/nfsen-ng/master/deploy/docker-compose.yml
nano docker-compose.yml
----

=== Обязательные изменения в `docker-compose.yml`

==== 1. Тег Docker-образа (важно)

В compose-файле по умолчанию указан `image: ghcr.io/mbolli/nfsen-ng:latest`, но тег **`latest` на GHCR сейчас отсутствует** (он появляется только у стабильных релизов без `-RC`). Замените строку `image`:

[source,yaml]
----
    image: ghcr.io/mbolli/nfsen-ng:edge
----

[IMPORTANT]
====
Для часового пояса *не UTC* (например `Europe/Moscow`, UTC+3) используйте **`edge`**, а не `1.0.0-RC.1`.

В `1.0.0-RC.1` есть баг: nfsen-ng парсит имена файлов `nfcapd.*` как UTC, хотя `nfcapd` на хосте пишет их в *локальном* времени. Симптомы:

* в Flows/Statistics команда `nfdump -R` уходит на **−3 ч** (выбрали 22:43 → в `-R` попало `19:40`);
* пустые графики и «No data available», хотя в терминале `nfdump -r ...` flows есть.

Исправление в upstream: link:https://github.com/mbolli/nfsen-ng/issues/145[#145] (май 2026, master / `edge`).
====

Альтернатива — зафиксированный RC (без фикса TZ):

[source,yaml]
----
    image: ghcr.io/mbolli/nfsen-ng:1.0.0-RC.1
----

[NOTE]
====
Если `docker compose pull` выдаёт `not found` для `:latest` — причина именно в теге. После смены на `edge` или `1.0.0-RC.1` pull заработает.
====

==== 2. Удалить переопределение `entrypoint` (важно)

В compose-файле с GitHub есть строка:

[source,yaml]
----
    entrypoint: ["/bin/bash", "/var/www/html/nfsen-ng/deploy/docker-entrypoint.sh"]
----

В образе GHCR (`1.0.0-RC.1`, `edge`) каталог `deploy/` *не копируется* — скрипт лежит в `/usr/local/bin/docker-entrypoint.sh` и уже прописан в образе. Переопределение из compose ломает запуск:

[source,text]
----
docker-entrypoint.sh: No such file or directory
exited with code 127
Restarting...
----

*Закомментируйте или удалите* всю строку `entrypoint:` у сервиса `nfsen`:

[source,yaml]
----
    # entrypoint: ["/bin/bash", "/var/www/html/nfsen-ng/deploy/docker-entrypoint.sh"]
----

==== 3. Открыть порт веб-интерфейса

Раскомментируйте блок `ports` у сервиса `nfsen`:

[source,yaml]
----
    ports:
      - "9000:9000"
----

Без этого веб-UI доступен только внутри Docker-сети.

==== 4. Указать источники (имена папок под `live/`)

Для двух endpoint-хостов:

[source,yaml]
----
      - NFSEN_SOURCES=win-pc01,ubuntu-pc01
----

[WARNING]
====
Значение `NFSEN_SOURCES` *должно совпадать* с именами каталогов:
`/var/nfdump/profiles-data/live/<имя>/`
====

==== 5. Порты для графиков RRD

[source,yaml]
----
      - NFSEN_PORTS=80,443,22,53
----

Список TCP/UDP-портов, по которым nfsen-ng строит отдельные RRD-графики.

==== 6. Часовой пояс (обязательно для MSK / UTC+3)

`nfcapd` на хосте Ubuntu создаёт файлы `nfcapd.YYYYMMDDHHMM` в *локальном* времени сервера (у нас — `Europe/Moscow`).

После фикса в `edge` (issue link:https://github.com/mbolli/nfsen-ng/issues/145[#145]) в `docker-compose.yml` у сервиса `nfsen` задайте:

[source,yaml]
----
      - TZ=UTC
      - NFCAPD_TZ=Europe/Moscow
----

[cols="1,2", options="header"]
|===
| Переменная | Назначение

| `TZ=UTC`
| Внутреннее время контейнера и PHP (как рекомендует upstream после переписывания TZ в 2026).

| `NFCAPD_TZ=Europe/Moscow`
| Часовой пояс, в котором *именуются* файлы `nfcapd.*` на хосте. Должен совпадать с `timedatectl` на коллекторе (`Europe/Moscow`).

|===

Проверка после `docker compose up -d --force-recreate`:

[source,bash]
----
date
timedatectl | grep "Time zone"
docker compose exec nfsen env | grep -E 'TZ|NFCAPD'
----

Ожидаемо на хосте: `Europe/Moscow`; в контейнере: `TZ=UTC`, `NFCAPD_TZ=Europe/Moscow`.

[WARNING]
====
*Не используйте* только `TZ=Europe/Moscow` без `NFCAPD_TZ` на образе `1.0.0-RC.1` — UI будет искать данные со сдвигом −3 ч, графики и Flows останутся пустыми.

После смены образа на `edge` и добавления `NFCAPD_TZ` выполните <<переимпорт-rrd,переимпорт RRD>> — старые RRD могли сохранить неверные метки времени.
====

=== Что можно не менять

[source,yaml]
----
    volumes:
      - /var/nfdump/profiles-data:/data/nfsen-ng
      - NFSEN_NFDUMP_PROFILES=/data/nfsen-ng
----

`NFSEN_NFDUMP_PROFILES` — путь *внутри контейнера*; volume связывает его с `/var/nfdump/profiles-data` на хосте.

=== Файл `settings.php`

Для стандартной установки *не требуется*. Достаточно переменных окружения в `docker-compose.yml`. Файл настроек нужен только для сложных сценариев (много фильтров, VictoriaMetrics и т.д.).

=== Пример фрагмента сервиса `nfsen` (проверено на Ubuntu 24.04, MSK)

Имя сервиса в compose — **`nfsen`** (контейнер при этом может называться `nfsen-ng`).

[source,yaml]
----
  nfsen:
    image: ghcr.io/mbolli/nfsen-ng:edge
    container_name: nfsen-ng
    restart: unless-stopped
    ports:
      - "9000:9000"
    volumes:
      - /var/nfdump/profiles-data:/data/nfsen-ng
    environment:
      - TZ=UTC
      - NFCAPD_TZ=Europe/Moscow
      - NFSEN_SOURCES=win-pc01,ubuntu-pc01
      - NFSEN_PORTS=80,443,22,53
      - NFSEN_NFDUMP_BINARY=/usr/local/nfdump/bin/nfdump
      - NFSEN_NFDUMP_PROFILES=/data/nfsen-ng
      # - NFSEN_LOG_LEVEL=INFO
    # entrypoint: НЕ переопределять — см. шаг 4.2
----

== Шаг 5. Запуск nfsen-ng

[source,bash]
----
cd ~/nfsen-ng
# убедитесь, что в compose указан тег edge (рекомендуется) или 1.0.0-RC.1, не latest
docker compose pull
docker compose up -d --force-recreate
----

Проверка:

[source,bash]
----
docker compose ps
docker compose logs -f nfsen
----

Остановка просмотра логов: `Ctrl+C` (контейнер продолжит работать).

Проверка, что контейнер видит файлы nfcapd:

[source,bash]
----
docker exec nfsen-ng ls /data/nfsen-ng/live/win-pc01/
docker exec nfsen-ng ls /data/nfsen-ng/live/ubuntu-pc01/
----

== Шаг 6. Первый вход и импорт данных

. Откройте браузер на сервере или с другого ПК в сети:

[source,text]
----
http://localhost:9000
http://<IP_СЕРВЕРА>:9000
----

. Перейдите на вкладку *Admin* (или *Settings → Import* в новых сборках `edge`).
. Нажмите *Initial Import* — импорт исторических nfcapd-файлов в RRD.
. Дождитесь завершения (время зависит от объёма данных).

Когда агенты уже шлют flow:

. *Admin* → *Trigger Import* — подхватить свежие файлы.
. Вкладка *Flows*: source `win-pc01`, интервал *1 hour*, timeline на время трафика → *Process data*.
. В строке команды `nfdump -R` должны быть метки **`2240…2315`**, а не `1940…` (MSK, после фикса TZ).

[TIP]
====
На вкладке Admin также доступен *Health check*: версия nfdump, пути к данным, статус daemon, *Capture freshness* по каждому источнику.

Во время *Force Rescan* индикатор *nfcapd* может быть красным, *daemon* — зелёным; это нормально. Красный *nfcapd* для `ubuntu-pc01` без агента — тоже ожидаемо.
====

[[переимпорт-rrd]]
=== Переимпорт RRD после смены образа или TZ

Если ранее использовали `1.0.0-RC.1` без `NFCAPD_TZ` или меняли часовой пояс:

[source,bash]
----
cd ~/nfsen-ng
docker compose up -d
docker compose exec nfsen rm -rf /var/www/html/nfsen-ng/backend/datasources/data/*
docker compose restart nfsen
----

Затем в UI: *Admin* → *Force Rescan* (дождаться *Rescan complete*) → *Trigger Import*.

Проверка в терминале (эталон, не зависит от UI):

[source,bash]
----
nfdump -R /var/nfdump/profiles-data/live/win-pc01/2026/07/03/nfcapd.202607032250:2026/07/03/nfcapd.202607032300 -c 10
----

[[clean-two-sources]]
=== Полный сброс данных двух тестовых источников

[WARNING]
====
Следующие команды необратимо удаляют raw flow-файлы `win-pc01` и `ubuntu-pc01`, а также построенные из них графики/RRD. Настройки nfcapd, Docker и nfsen-ng сохраняются. Перед выполнением остановите оба endpoint-агента, чтобы во время очистки не приходили новые datagram.
====

На Windows endpoint (PowerShell от администратора):

[source,powershell]
----
Stop-Service netflowAgent
----

На Linux endpoint:

[source,bash]
----
sudo systemctl stop netflowAgent
----

На коллекторе сначала убедитесь, что nfsen-ng настроен только на эти два тестовых источника:

[source,bash]
----
cd ~/nfsen-ng
docker compose exec nfsen env | grep NFSEN_SOURCES
----

Остановите приём и удалите контейнер nfsen-ng вместе с его внутренним RRD cache. `docker compose down` без `-v` не удаляет host-каталог `/var/nfdump/profiles-data`:

[source,bash]
----
sudo systemctl stop nfcapd-win-pc01.service nfcapd-ubuntu-pc01.service
cd ~/nfsen-ng
docker compose down
----

Необязательная резервная копия перед удалением:

[source,bash]
----
sudo tar -C /var/nfdump/profiles-data/live \
  -czf "$HOME/netflow-before-reset-$(date +%Y%m%d-%H%M%S).tar.gz" \
  win-pc01 ubuntu-pc01
----

Удалите и пересоздайте только два source-каталога:

[source,bash]
----
sudo rm -rf -- \
  /var/nfdump/profiles-data/live/win-pc01 \
  /var/nfdump/profiles-data/live/ubuntu-pc01
sudo mkdir -p \
  /var/nfdump/profiles-data/live/win-pc01 \
  /var/nfdump/profiles-data/live/ubuntu-pc01

sudo systemctl start nfcapd-win-pc01.service nfcapd-ubuntu-pc01.service
cd ~/nfsen-ng
docker compose up -d
----

Контроль чистого состояния:

[source,bash]
----
find /var/nfdump/profiles-data/live/win-pc01 -type f
find /var/nfdump/profiles-data/live/ubuntu-pc01 -type f
sudo systemctl --no-pager status nfcapd-win-pc01.service nfcapd-ubuntu-pc01.service
docker compose ps
sudo ss -ulnp | grep -E ':9995|:9996'
----

До повторной установки endpoint-агентов первые две команды не должны показывать `nfcapd.*`. После появления новых flow в UI выполните *Admin* → *Force Rescan* → *Trigger Import* или дождитесь автоимпорта.

== Шаг 7. Автозапуск nfcapd (systemd)

Для двух источников создайте два unit-файла.

*win-pc01* — `/etc/systemd/system/nfcapd-win-pc01.service`:

[source,ini]
----
[Unit]
Description=nfcapd collector for win-pc01
After=network.target

[Service]
Type=forking
ExecStartPre=/bin/mkdir -p /var/nfdump/profiles-data/live/win-pc01
ExecStart=/usr/bin/nfcapd -w /var/nfdump/profiles-data/live/win-pc01 -z=lz4 -S 1 -p 9995 -D
Restart=on-failure

[Install]
WantedBy=multi-user.target
----

*ubuntu-pc01* — `/etc/systemd/system/nfcapd-ubuntu-pc01.service`:

[source,ini]
----
[Unit]
Description=nfcapd collector for ubuntu-pc01
After=network.target

[Service]
Type=forking
ExecStartPre=/bin/mkdir -p /var/nfdump/profiles-data/live/ubuntu-pc01
ExecStart=/usr/bin/nfcapd -w /var/nfdump/profiles-data/live/ubuntu-pc01 -z=lz4 -S 1 -p 9996 -D
Restart=on-failure

[Install]
WantedBy=multi-user.target
----

Включите оба сервиса:

[source,bash]
----
sudo systemctl daemon-reload
sudo systemctl enable --now nfcapd-win-pc01.service nfcapd-ubuntu-pc01.service
sudo systemctl status nfcapd-win-pc01.service nfcapd-ubuntu-pc01.service
----

Контейнер nfsen-ng уже имеет `restart: unless-stopped` в compose — перезапускается вместе с Docker.

== Шаг 8. Настройка endpoint-агентов (netflowAgent)

Агент link:netflowAgent-design.adoc[netflowAgent] на каждом хосте отправляет NetFlow v9 на коллектор. IP сервера — из шага 1 (`ip -4 addr`).

* link:netflowAgent-install-windows.adoc[Установка на Windows (win-pc01)] — **Npcap** + агент + служба
* link:netflowAgent-install-linux.adoc[Установка на Linux (ubuntu-pc01)] — **libpcap** + агент (в разработке)

[cols="1,1,1,1", options="header"]
|===
| Хост | IP коллектора | UDP-порт | Каталог на коллекторе

| win-pc01
| `<IP_коллектора>`
| 9995
| `live/win-pc01/`

| ubuntu-pc01
| `<IP_коллектора>`
| 9996
| `live/ubuntu-pc01/`

|===

[WARNING]
====
Порт в агенте *должен совпадать* с портом `nfcapd` для этого хоста. Иначе данные попадут не в ту папку или не попадут вообще.
====

`nfcapd` принимает NetFlow v5/v9 и IPFIX. После появления flow-файлов nfsen-ng подхватит данные при следующем импорте или через встроенный import daemon.

== Устранение неполадок

[cols="1,2", options="header"]
|===
| Симптом | Решение

| Пустые графики, файлы nfcapd есть
| См. <<tz-empty-ui,сдвиг часового пояса в UI>>; переимпорт RRD; образ `edge`

| Нет файлов nfcapd на диске
| Проверьте агенты, firewall, `find ... nfcapd.*`, `tcpdump -ni any udp port 9995`

| Flows: «No data available», в терминале flows есть
| Сдвиг −3 ч в `-R`; `edge` + `NFCAPD_TZ`; <<переимпорт-rrd,переимпорт RRD>>

| Источник не виден в UI
| `NFSEN_SOURCES` не совпадает с именем папки `live/<source>`

| `http://IP:9000` не открывается
| Раскомментируйте `ports: "9000:9000"`, проверьте `ufw`

| `docker compose pull`: `latest: not found`
| Замените `image: ...:latest` на `:edge` (рекомендуется) или `:1.0.0-RC.1`

| Контейнер `Restarting (127)`, `docker-entrypoint.sh: No such file`
| Закомментируйте строку `entrypoint:` в compose (см. шаг 4.2)

| `permission denied` при Docker
| Выполните `sudo reboot` после `usermod -aG docker`; проверьте `groups`

| `service "nfsen" is not running` при `docker compose exec`
| Сначала `docker compose up -d`; имя сервиса — `nfsen`, не `nfsen-ng`

| SSE / графики не обновляются
| Не сжимайте прокси-трафик к nfsen-ng; для Caddy см. wiki nfsen-ng

|===

[[tz-empty-ui]]
=== Сдвиг часового пояса в UI (MSK, UTC+3)

Типичные симптомы на `1.0.0-RC.1`:

* timeline показывает 22:43, а в Flows команда `nfdump -R` содержит `...1940:...2015` (−3 ч);
* LIVE / RRD со сдвигом (+3 ч или год 2024 вместо 2026);
* ползунок timeline не уходит в будущее (упирается в «сейчас») — это нормально.

*Решение* (проверено в проекте netflowAgent, июль 2026):

. Образ `ghcr.io/mbolli/nfsen-ng:edge`
. `TZ=UTC` и `NFCAPD_TZ=Europe/Moscow` в `environment`
. `docker compose up -d --force-recreate`
. <<переимпорт-rrd,Переимпорт RRD>> + *Force Rescan* + *Trigger Import*
. Flows: в `-R` метки **`22xx`**, в таблице появляются flows (например 50 flows)

Upstream: link:https://github.com/mbolli/nfsen-ng/issues/145[#145], link:https://github.com/mbolli/nfsen-ng/issues/103[#103], link:https://github.com/mbolli/nfsen-ng/issues/68[#68] (TZ браузера / клиента).

[NOTE]
====
Имена файлов `nfcapd.*` округляются до *5 минут* (`22:43` → `2240` в `-R`). Это нормально.
====

[[flows-table-utc]]
==== Время в таблице Flows (−3 ч от MSK)

После настройки `edge` + `NFCAPD_TZ=Europe/Moscow` *timeline* и строка `nfdump -R` в Flows *совпадают* с выбранным интервалом (например выбрали 14:55–16:00 → в `-R` `...1455:...1600`).

Колонки *First*, *Last*, *Received* в таблице при этом могут показывать время на **3 часа раньше** (UTC), чем ожидается в MSK:

[cols="1,1,1", options="header"]
|===
| Где | Пример | Пояснение

| Timeline / `-R`
| `14:55 – 16:00`
| Имена файлов nfcapd (MSK) — *верно*

| First / Last в таблице
| `11:54`
| UTC; **+3 ч = 14:54 MSK** — тот же flow

| Graphs
| ось времени
| Обычно совпадает с ожиданиями; ориентир для анализа

|===

Это *косметика UI* nfsen-ng / nfdump, не ошибка агента и не потеря данных. Для точного времени в терминале: `nfdump -R ... -c 10` (там тоже UTC — прибавьте +3 ч для MSK).

Проверено: link:https://github.com/kolixxx/nflowCapAgent[nflowCapAgent], июль 2026 — графики обновляются автоматически, flows находятся.

[[auto-import]]
=== Автоимпорт: нужен ли Trigger Import каждый раз?

*Нет.* В штатном режиме ручной импорт не нужен.

[source,text]
----
netflowAgent → nfcapd (файлы каждые ~5 мин)
                    ↓
         import daemon nfsen-ng (Watching N dirs)
                    ↓
              RRD → Graphs / Statistics
----

* *Import daemon* (Admin: *Watching 2 dirs*, *Last auto import: N min ago*) подхватывает новые `nfcapd.*` и обновляет RRD примерно каждые **5 минут**.

[cols="1,2", options="header"]
|===
| Действие | Когда

| Ничего не нажимать
| Агент работает, Graphs догоняют сами (задержка ~5 мин)

| *Trigger Import*
| Нужны данные «прямо сейчас», не хотите ждать auto-import

| *Force Rescan*
| Смена TZ/образа, сброс RRD, новый источник, «всё сломалось» — *не* на каждый новый flow

|===

Если Graphs не растут 15+ мин при работающем агенте: Admin → *Capture freshness* для `win-pc01`; один раз *Trigger Import*; проверить, что агент на Windows не остановлен.


[source,bash]
----
# Логи nfsen-ng
docker compose -f ~/nfsen-ng/docker-compose.yml logs -f nfsen

# Перезапуск (после смены environment — лучше --force-recreate)
cd ~/nfsen-ng && docker compose up -d --force-recreate

# Сброс RRD и переимпорт (после смены TZ / образа на edge)
docker compose exec nfsen rm -rf /var/www/html/nfsen-ng/backend/datasources/data/*
docker compose restart nfsen
# затем в UI: Force Rescan → Trigger Import

# Статус nfcapd
sudo systemctl status nfcapd
----

[[тест-softflowd]]
== Приложение: тест через softflowd (без netflowAgent)

Пока агенты netflowAgent не готовы, можно *локально* проверить цепочку «отправка → nfcapd → файлы на диске» на самой VM-коллекторе:

[source,bash]
----
sudo apt install -y softflowd
ip link   # найдите интерфейс, например enp0s3 или wlp2s0

sudo softflowd -i enp0s3 -v 9 -n 127.0.0.1:9996
----

Подставьте свой интерфейс. Через несколько минут проверьте:

[source,bash]
----
find /var/nfdump/profiles-data/live/ubuntu-pc01 -name "nfcapd.*" -ls
----

Затем выполните *Initial Import* в веб-интерфейсе nfsen-ng.
