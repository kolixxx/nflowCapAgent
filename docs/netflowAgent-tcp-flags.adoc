= netflowAgent: сбор TCP-флагов
:toc:
:toclevels: 2

== Кратко

Агент использует стандартный механизм NetFlow/IPFIX:

. Npcap на Windows или libpcap на Linux передаёт агенту сетевой кадр.
. Агент находит IPv4- и TCP-заголовки.
. Из каждого TCP-пакета читаются стандартные TCP flags.
. Для всех пакетов одного flow выполняется побитовое OR.
. Результат передаётся как NetFlow v9 `TCP_FLAGS` или IPFIX IE 6 `tcpControlBits`.

Это общепринятый способ, а не собственный формат netflowAgent.

== TCP-флаги в пакете

Классические TCP-флаги находятся в байте со смещением 13 от начала TCP-заголовка:

[source,text]
----
Бит:    7    6    5    4    3    2    1    0
Флаг:  CWR  ECE  URG  ACK  PSH  RST  SYN  FIN
----

[cols="1,1,2", options="header"]
|===
| Флаг | Значение | Назначение

| FIN | `0x01` | Нормальное завершение соединения
| SYN | `0x02` | Установка соединения
| RST | `0x04` | Принудительный сброс
| PSH | `0x08` | Передача данных приложению
| ACK | `0x10` | Подтверждение
| URG | `0x20` | Срочные данные
| ECE | `0x40` | ECN Echo
| CWR | `0x80` | Congestion Window Reduced
|===

== Разбор пакета

Агент вычисляет длину IPv4-заголовка через `IHL`, проверяет `protocol` и находит начало TCP:

[source,rust]
----
let ihl = (version_ihl & 0x0f) as usize * 4;
let protocol = data[9];
let l4 = &data[ihl..total_length];

let tcp_flags = if protocol == IPPROTO_TCP {
    parse_tcp_flags(l4)
} else {
    0
};
----

Флаги читаются из стандартного байта 13:

[source,rust]
----
fn parse_tcp_flags(tcp: &[u8]) -> u8 {
    if tcp.len() >= 14 {
        tcp[13]
    } else {
        0
    }
}
----

Для UDP значение равно нулю: в UDP TCP-флагов нет.

== Накопление флагов внутри flow

Flow определяется однонаправленным 5-tuple:

[source,text]
----
source IP
destination IP
source port
destination port
protocol
----

Обратное направление соединения является отдельным flow.

Для каждого пакета агент выполняет cumulative bitwise OR:

[source,rust]
----
entry.tcp_flags |= tcp_flags;
----

Пример:

[source,text]
----
SYN       0x02  00000010
ACK       0x10  00010000
PSH+ACK   0x18  00011000
FIN+ACK   0x11  00010001
                 ──────── OR
Результат 0x1B  00011011  SYN + ACK + PSH + FIN
----

Поле показывает, какие флаги встречались хотя бы в одном пакете flow. Оно не сохраняет порядок пакетов и количество появлений каждого флага.

== Экспорт IPFIX

В IPFIX используется стандартный Information Element:

[source,text]
----
IE:     6
Имя:   tcpControlBits
Тип:   unsigned16
Длина: 2 байта
----

Template 259 объявляет поле как `(6, 2)`, а восьмибитная маска агента передаётся в младших битах `u16`.

Например:

[source,text]
----
06 001b
│  └──── tcpControlBits = 0x001B
└─────── protocolIdentifier = 6 (TCP)
----

`0x001B` означает `SYN + ACK + PSH + FIN`.

В NetFlow v9 используется стандартное поле type 6 `TCP_FLAGS`.

== Отображение в nfdump

`nfdump` отображает флаги в порядке:

[source,text]
----
C E U A P R S F
----

Например:

[source,text]
----
...AP.SF
----

означает `ACK + PSH + SYN + FIN`. Точка означает, что соответствующий флаг в этом flow не наблюдался.

== Ограничения агрегирования

По итоговой маске нельзя определить:

* порядок TCP-пакетов;
* количество SYN/ACK/FIN;
* пакет, в котором появился конкретный флаг;
* полную историю TCP handshake.

Для этого требуется PCAP. NetFlow/IPFIX хранит агрегированную информацию.

Длинное соединение также может разделяться по тайм-аутам:

[source,text]
----
inactive_timeout_secs = 15
active_timeout_secs   = 60
----

Поэтому первая запись может содержать SYN, промежуточная — только ACK/PSH, а последняя — FIN или RST.

== Ограничение текущей версии

Агент v0.3.1 собирает восемь классических флагов:

[source,text]
----
CWR ECE URG ACK PSH RST SYN FIN
----

Девятый TCP-бит `NS` находится в предыдущем байте заголовка и сейчас не собирается. Для nfdump и большинства сценариев NetFlow это обычное поведение; при необходимости поддержку `NS` можно добавить отдельно.
